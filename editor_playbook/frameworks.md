# 編集長のフレームワーク辞典

引き出しとして使える方法論集。状況別に呼び出す。

## 戦略・ロードマップ

### Goal Oriented ロードマップ（5ステップ）
1. **Ultimate Goal設定** — 1on1総研の3年Goal（記事×訪問者×リード×商談）
2. **ギャップ分析** — 現状とのデルタを定量化
3. **ゴール設定** — ユーザー視点／競合視点／市場視点の3軸
4. **施策評価** — どの施策がどのゴールにどの強度で寄与するか
5. **ロードマップ化** — Goal起点で四半期テーマを並べる

### Profit Tree
全KPIを利益への寄与で分解。「読者数×CVR×LTV」→各記事/施策がどのノードに効くかタグ付け。四半期で更新。

### サービスコンセプト階層（0→1の型）
```
ビジョン
  └─ ターゲットの課題/インサイト
      └─ コンセプト（20字以下のUVP）
          └─ 具体策
```

### 競合優位性の二層分析
- **短期差別化（〜1年）**: トレンド記事・先行配信・SNS活性化
- **中期MOAT（3年）**: ノウハウ蓄積・コミュニティ・信頼ネットワーク

## グロース・改善

### ARRRA（AARRRの正しい優先順）
A**ctivation → R**etention → R**eferral → R**evenue → **A**cquisition の順で最大化。「PV取ってから品質」を逆転。

### 8ステップ・グロースサイクル
①KGI設定 → ②現状分析 → ③課題特定 → ④仮説立案 → ⑤施策設計 → ⑥実行 → ⑦計測 → ⑧学習・改訂

### 因果関係の3パターン
施策設計前に必ず分類:
1. **正しい因果関係** — Aを増やせばBが増える
2. **逆相関** — 結果系を原因と勘違い
3. **第三要因による相関** — 共通の原因が両者を動かしている

### スランプ時の打ち手
KPI低迷時の本質理解不足が原因 → 数値分析×記者インタビュー（執筆背景）の組合せで仮説3つ立てる

## 検証・PMF

### 12人ルール（PMF検証4ステップ）
1. **課題発見** — 12人にインタビュー、共通課題を抽出
2. **解決検証** — 12人にコンセプトを当てて反応取得
3. **アーリーアダプター獲得** — 5人にコミットを取り付け
4. **購買意思検証** — 実際にお金/時間を払うか

### ユーザーテストの作法
施策サイクルに組み込むと成功率が10倍→8倍に跳ね上がる。3〜5名で十分。Google Form＋簡単なヒアリング。

### 仮説/データドリブン企画
全企画に事前数値仮説（予想PV/滞在時間/CVR）→実績照合。優れたチームは全議論がデータと仮説起点。

## UX設計

### UX 5レイヤー（Garrett）
**Strategy → Scope → Structure → Skeleton → Surface**

特に **Structure**（情報構造と遷移設計）は軽視されがち。記事公開前に IA Mapping と User Flow Diagram を1枚描く。

### 最高/最悪フロー
- **最高シナリオ**: 最短で目的記事に到達する読者体験
- **最悪シナリオ**: 情報迷子になる読者体験

両方を CJM で描かないとギャップが定量化できない。90%の企業が両方定義できていない。

### UX方程式
良い記事と悪い記事を3本ずつ抽出 → 共通要素を方程式化 → 判定基準として運用。

### オンボーディング（AIサービス的な発想）
新規読者向けに「最初の3記事ナビゲーション」を設計:
- HR基礎 → 事例 → 実装ガイド
- 初訪問者が迷わず価値を感じる道筋
- SNS告知時に「最初に読むべき記事」シリーズ化

### 複雑性最適化
ユーザーは機能の20%しか使わない。アクセス解析から「実際に使われるカテゴリ・タグ」を抽出し、不要なナビ要素を削除。

## 組織・育成

### 育成6領域
1. 思考法・心構え
2. 仕事の基礎スキル
3. プロダクトマネジメント（何を・なぜ・どう作るか）
4. プロジェクトマネジメント（誰が・いつまでに・どう作るか）
5. サービスデザイン（UX設計）
6. グロース（事業成長駆動）

→ 編集チームに翻訳: 思考法／基礎／企画(PdM)／進行管理(PjM)／読者体験設計／流通

### 意思決定3要素
- **芯（こころ）** — 読者軸の判断基準
- **しなやかさ** — トレンドへの対応力
- **見切り** — 公開ラインの明示

各編集者に権限委譲設計を1枚で。

### 3つのRetro（編集版）
1. **記事単位Retro** — 公開後48時間以内、3行（届いたか／タイトル適切か／次への学び）
2. **週次編集進行Retro** — フローのボトルネック共有
3. **月次チーム関係性Retro** — モチベ・関係性の振り返り

1on1総研の編集自体が1on1の実践現場になる。

### 事例ストック運用
Slackチャネル `#事例ストック` で週1回以上、秀逸な記事構成例をシェア。3ソース推奨（競合/海外/SNS）。

## マーケ・グロース戦術

### カテゴリ啓蒙の3ステップ（BtoB）
1. **概念啓蒙** — カテゴリの世界観を展開
2. **市場教育** — ビジネス上の必要性を啓発
3. **第一想起確保** — カテゴリ＝企業の認識を定着

→ 「1on1といえば1on1総研」を作る。記事は企業宣伝ではなくカテゴリ啓蒙に注力（直近10本で50%以上）。

### 初期ユーザー獲得（資金なし前提）
- 有料広告は不向き
- オーガニック拡散・企業提携・PR・SNS口コミ
- 既読ユーザーの拡散をKPIに

### 自己破壊戦略（AI時代）
- 最悪シナリオ特定 → 破壊コスト低減 → 複数事業の並行運営
- 競合のAIサービスを月1回監視
- 「もしAIが1on1総研の領域に来たら」のシナリオプラン

## AI活用（編集業務）

### 新規事業立ち上げ的AI活用
- **市場調査**: ChatGPT Deep Research（X・Yahoo知恵袋からインサイト抽出）
- **競合分析**: 国際事例のフィルタリング
- **STP**: セグメンテーション・ターゲティング・ポジショニング戦略
- **プロト**: ビジュアル・機能プロトタイピング

→ 新連載・特集の市場調査をDeep Researchで自動化、編集工数30%削減を目指す

### AI実装の80/20則
80%の労力はチューニング・精度調整（90%精度を基準）。AI執筆支援ツール導入時は「ガイドラインの整理・品質チェックプロセス」に労力を寄せる。
